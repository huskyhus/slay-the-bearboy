# Slay the Bearboy 企画書

## 1. プロジェクト概要

**Slay the Bearboy** は、オリジナルLINEスタンプ「ハスくんと一緒だよ」のキャラクター「ハスくん」を主人公とした、Slay the Spire 風のシングルプレイヤー・デッキ構築型カードゲームである。本ドキュメントは、3人チームによる開発の v1.0（プロトタイプ）に向けた企画・技術方針書である。

### 1.1 想定プレイヤー

- LINEスタンプ「ハスくんと一緒だよ」のファン層。
- Slay the Spire 系のデッキ構築型カードゲームに親しんだプレイヤー層。
- ブラウザで気軽に遊べる小規模ゲームを好む層。

### 1.2 リファレンス作品

- **Slay the Spire**：戦闘システム、デッキ構築、カード・レリック・敵の構造を主に参考にする。
- **Cookie Clicker**：プロトタイプ段階の技術構成として「サーバー不要・ブラウザ完結」のスタイルを参照する。

---

## 2. v1.0 (プロトタイプ) スコープ

### 2.1 ゴール

「1ステージを通してプレイできる最小構成のゲーム体験」を提供する。マップ、レリック、複数戦闘、永続的なメタプログレッションなどの要素は v1.0 では含めない。

### 2.2 含めるゲーム要素

- **プレイヤー**：HP、ブロック（防御値）、エナジー（毎ターン回復するコスト資源）を持つ。主人公はハスくん。
- **敵**：1〜3体程度。各敵はHPと「次のアクション」表示を持ち、ターンごとに攻撃または防御などを行う。
- **デッキシステム**：以下の4ゾーンを管理する。
  - デッキ（山札）
  - 手札
  - 捨て札
  - 除外
- **カード**：10〜15種類程度。
  - 攻撃カード（複数種）
  - 防御カード（複数種）
  - スキルカード（最低1〜2種）
- **ターン制バトルループ**：
  1. プレイヤーターン開始：エナジー回復、ドロー
  2. プレイヤーがカードをプレイ
  3. ターン終了
  4. 敵の行動
  5. 次のプレイヤーターンへ
- **勝敗判定**：プレイヤーHPが0で敗北、すべての敵を撃破で勝利。
- **結果画面**：勝敗の表示と簡単なリザルト。

### 2.3 v1.0 で含めないもの（v1.1 以降）

- マップ画面・複数戦闘の連続
- レリック（永続効果のアイテム）
- 戦闘後のカード報酬選択
- ショップ・休憩所などのイベントマス
- セーブ/ロード機能
- BGM・効果音（任意で v1.0 に含める可能性あり）
- バックエンドサーバ

## 3. 技術構成

### 3.1 全体方針

- プロトタイプ段階では **バックエンドを持たない、完全にフロントエンドのみで動作するクライアントサイドアプリケーション** として構築する（Cookie Clicker のような構成）。
- ただし将来的にバックエンドサーバを追加する可能性を考慮し、フレームワーク選定では拡張性を担保する。
- 言語は **TypeScript のみ** を使用する。Python など他の言語は補助ツールとしても本プロジェクトでは使用しない。

### 3.2 採用スタック

| 領域 | 技術 | 選定理由 |
|------|------|----------|
| 言語 | TypeScript | 型安全性、チーム全員が習熟 |
| フレームワーク | Next.js (App Router) | Vercel デプロイが容易、将来バックエンドを追加する際に同一フレームワーク内で対応可能 |
| スタイル | Tailwind CSS | 高速なプロトタイピング |
| 状態管理 | Zustand | 軽量、ゲームステートの管理に向いている |
| アニメーション | Framer Motion（必要に応じて導入） | カード演出の実装が容易 |
| ホスティング | Vercel | Next.js との親和性、`git push` で自動デプロイ |
| Lint/Format | ESLint + Prettier | 標準的なコード品質維持 |

### 3.3 アーキテクチャ原則

**最重要原則：`src/game/` ディレクトリ内のコードは DOM/React/Next.js に依存しない純粋なロジックとする。**

この原則の意図：

- ゲームロジック（カード効果、戦闘計算、敵AI、状態遷移）を UI から完全に分離する。
- 将来的にシミュレータ（バランス調整用のバッチ実行）やバックエンドサーバへの移植が必要になった場合に、`src/game/` をそのまま流用できる。
- テスタビリティを確保する（純粋関数は単体テストが容易）。

`src/store/` は Zustand ストアであり、`src/game/` の純粋関数を呼び出して状態を更新する「薄いラッパー」として機能する。`src/ui/` は React コンポーネントを置き、`src/store/` を購読して描画する。

### 3.4 ディレクトリ構成（v1.0 時点）

リポジトリ全体は関心事ごとに分割している。

```
slay-the-bearboy/
├── web/                    # Next.js プロジェクト一式
│   ├── app/                # Next.js App Router（ページ）
│   ├── src/
│   │   ├── game/           # 純粋ロジック（DOM/React非依存）
│   │   ├── ui/             # React コンポーネント
│   │   └── store/          # Zustand ストア
│   ├── data/               # カード定義（JSON）
│   ├── public/             # Next.js 静的配信ディレクトリ
│   ├── stickers/           # ハスくん LINE スタンプ素材（原本）
│   └── package.json
├── compose.yaml            # ローカル dev 用 Docker Compose（本番は Vercel）
├── docs/                   # 企画書・仕様などのドキュメント
├── Taskfile.yml            # 起動・停止などのショートカット
└── .github/                # Issue/PR テンプレート、Actions 等
```

ゲームロジックは `web/src/game/` に置き、UI (`web/src/ui/`) や状態管理 (`web/src/store/`) とは独立に保つ。`tools/`, `web/src/sim/` などは v1.0 では設けず、必要が出てきた時点で追加する。

### 3.5 Next.js 利用上の注意

- ゲーム本体（バトル画面など、状態を持つインタラクティブ画面）は **クライアントコンポーネント（`'use client'`）** として実装する。
- Zustand ストアはクライアント側のみで動作するため、SSR でハイドレーションミスマッチを起こさないよう注意する。
- タイトル画面など静的な画面はサーバーコンポーネントでも可。
- ルーティング例：
  - `/`（タイトル画面）：サーバーコンポーネント
  - `/play`（ゲーム本体）：クライアントコンポーネント

### 3.6 デプロイと Docker の役割分担

- **本番デプロイは Vercel に確定**。`git push` での自動デプロイを利用する。
- Next.js プロジェクトはリポジトリ直下ではなく `web/` 配下にあるため、**Vercel のプロジェクト設定で Root Directory を `web` に指定する**（モノレポ向けの標準機能）。これによりサブディレクトリ構成のままデプロイできる。
- **Docker（リポジトリ直下の `compose.yaml`）はローカル開発専用**。Vercel は本番ビルドで Docker を一切参照せず、独自パイプラインで `next build` を実行する。したがって Docker 設定は本番に影響しない。
- Docker はローカル dev 専用と割り切るため、カスタムイメージは作らず公式 `node` イメージを直接使い、`web/` のソースはバインドマウントで提供する。Dockerfile も `.dockerignore` も持たない。
- ポートは 3000 に固定する（プロトタイプでは可変にする必要がないため）。

## 4. チーム体制

3人チーム。役割は以下のように分担する想定。

| 役割 | 主担当領域 |
|------|------------|
| メンバーA：フロントエンドUI担当 | React コンポーネント、画面構成、Tailwind スタイリング、Framer Motion による演出 |
| メンバーB：ゲームロジック担当 | `src/game/` の純粋ロジック実装、カード効果、戦闘システム、敵AI、Zustand ストア設計 |
| メンバーC：コンテンツ・統合担当 | カードデータ定義（JSON）、ハスくん画像の取り込みと配置、README やドキュメント整備、フロントとロジックを繋ぐ部分の調整 |

役割は完全分離ではなく、Issue 単位での協力を前提とする。
